# v0.3.0 troubleshooting — `ModuleNotFoundError: zeroconf` on production

If the add-on crashes on startup with:

```
File "/app/gateway/main.py", line 16, in <module>
    from gateway.ha_discovery import HaDiscoveryAdvertiser, HaDiscoveryConfig
  File "/app/gateway/ha_discovery.py", line 33, in <module>
    from zeroconf.asyncio import AsyncServiceInfo, AsyncZeroconf
ModuleNotFoundError: No module named 'zeroconf'
```

the add-on image at HA Supervisor does not contain the `zeroconf`
package, even though `requirements-gateway.txt` lists it and the
Dockerfile installs it.

## Likely cause

HA Supervisor's local image cache is serving a stale image with the
same tag (`ghcr.io/markminnoye/ipbuilding-gateway:0.3.0`) but built
before `zeroconf` was added to the requirements. The build itself
succeeded; only the cached image on the HA host is out of date.

## How to confirm

SSH into the Home Assistant host and force a fresh image pull:

```bash
docker pull ghcr.io/markminnoye/ipbuilding-gateway:0.3.0
docker inspect ghcr.io/markminnoye/ipbuilding-gateway:0.3.0 \
  | grep -E '"(Id|Parent|Created)"'
```

Then check whether the freshly-pulled image actually has zeroconf:

```bash
docker run --rm ghcr.io/markminnoye/ipbuilding-gateway:0.3.0 \
  pip show zeroconf
```

Expected: package metadata (name, version, location). If the
`pip show` returns "package not found", the published image is
genuinely missing the dependency — escalate to "rebuild forced".

If `pip show` returns the package, the local Supervisor cache is
the cause. Proceed with the reinstall step below.

## How to reinstall cleanly

In HA Supervisor:

1. **Settings → Add-ons → IPBuilding Gateway → ⋮ → Uninstall**
2. **Settings → Add-ons → Add-on store → ⋮ → Repositories → Refresh**
3. **Settings → Add-ons → IPBuilding Gateway → Install**
   (the new image will be pulled on first start)

If the add-on store still shows v0.1.4 as the latest available
version after the repository refresh, the new image has not been
picked up by the store cache. In that case:

1. **Settings → Add-ons → IPBuilding Gateway → ⋮ → Update** (force pull)
2. Or via SSH:
   ```bash
   ha addons reload local_ipbuilding_gateway
   ha addons install local_ipbuilding_gateway
   ```

## Verifying the image content

Once the add-on starts, check the add-on log. The first lines should
include the new HA-discovery startup:

```
INFO  gateway.ha_discovery  Starting HA discovery  instance_id=...  base_url=...  addon=...
INFO  gateway.ha_discovery  Zeroconf registered: ipbgw-XXXXXXXX._ipbgw._tcp.local. on port 8080
```

If the `Starting HA discovery` line is missing, the add-on is still
running the old image — repeat the uninstall / reinstall cycle.

## Why a `pip install` fallback is not added to the runtime

The container's entry point is `run.sh`, which exports environment
variables and then `exec python -m gateway`. A `pip install zeroconf`
in `run.sh` would silently mask a broken image (e.g. a future dependency
typo or version-incompatibility) and hide the underlying image-cache
mismatch from the operator. The image-cache mismatch is a deployment
issue, not a runtime one — fixing it at startup would obscure the
real signal and make the next broken-image failure silent.

## When this is *not* the cause

If `pip show zeroconf` returns the package but the add-on still
crashes on startup with the same traceback:

- Check the image SHA matches `c47f3c4` (the v0.3.0 release commit):
  `docker inspect --format='{{index .Config.Labels "org.opencontainers.image.revision"}}' …`
- If the SHA is older, the build context was stale; force a rebuild
  by deleting and re-creating the tag.

## Prevention

To reduce the chance of this recurring:

- After tagging a release, give the GitHub Actions build ~3 minutes
  to complete before installing the add-on in production.
- The build status is visible at
  github.com/markminnoye/IPBuilding-Gateway/actions.
