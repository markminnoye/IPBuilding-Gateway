## 3. Why the `_ipbgw._tcp.local.` rename?

`zeroconf 0.149` enforces **RFC 6763 §7.2** via a strict validator:
the service-type label after the leading underscore must be ≤ 15
bytes. The original `_ipbuilding-gateway._tcp.local.` has an 18-byte
label and is rejected at registration time. Renamed to `_ipbgw`
(5-byte label) — also a shorter and more conventional prefix.

Both gateway and companion must use the exact same string. Tests
enforce this: `tests/test_ha_discovery.py::test_service_type_constant`
and `tests/test_config_flow_parsing.py::test_zeroconf_service_type_label_within_rfc_limit`.

---

## 4. Why the SRV-vs-TXT fix?

HA's `ZeroconfServiceInfo` is built by
`homeassistant.components.zeroconf.info_from_service()`:
```python
return ZeroconfServiceInfo(
    ip_address=ip_address,
    ip_addresses=ip_addresses,
    port=service.port,        # ← from SRV record, not TXT
    hostname=service.server,
    type=service.type,
    ...
)
```
The `port` and `host` attributes come from the SRV record. The TXT
record is for arbitrary metadata (we use it for `instance_id`,
`base_url`, `homeassistant_addon`, `schema_version`, `version`).

The previous companion parser read `properties["host"]` /
`properties["port"]` from the TXT and aborted with
`Invalid zeroconf payload` when those keys were absent. Now it takes
the SRV-level values as primary and falls back to TXT for back-compat.

The regression test `test_srv_host_port_takes_precedence_over_txt`
pins the precedence rule.

## 5. Why `metadata_timeout_s` is configurable?

The default 2 s timeout in `ModuleMetadataCache.refresh` was too tight
on the user's VLAN. Raised to 5 s and made configurable via
`GATEWAY_METADATA_TIMEOUT_S` env / add-on option, so operators can
tune it without code changes.

## 6. Why bounded-concurrency in `getSysSet`?

Three modules were fanned out with `asyncio.gather` over a default
`aiohttp.ClientSession()` (no connector limit). On a busy boot moment
(ARP sweep + mDNS broadcast + Supervisor POST in parallel), the IP1100PoE
(input module) would sometimes lose the connection race and time out
even though the same module responded in < 1 s when queried directly
with `curl`. Adding `asyncio.Semaphore(3)` plus a `TCPConnector(limit=8)`
gives predictable behaviour.

## 7. Things to look for in review

1. **Threading/event-loop safety** of the new `Semaphore(3)` and the
   `AsyncZeroconf()` lifecycle (`start`/`stop`). Both run in the
   gateway's main asyncio loop.
2. **Error handling** in `HaDiscoveryAdvertiser._hassio_announce_loop`:
   backoff doubles up to 60 s, restart on every re-announce after 5
   minutes. Is the backoff right for transient Supervisor outages?
3. **Race between `stop()` and the announce loop**: `stop()` cancels
   the task and closes the HTTP session, but the loop might be mid-
   `DELETE /discovery/{uuid}`. Is the ordering safe?
4. **Schema-versioning**: `DISCOVERY_SCHEMA_VERSION = 1`. The TXT
   parser allows `0` for backward-compat. What should happen when
   the gateway upgrades to schema 2?
5. **Sticky `instance_id`**: written to `{data_dir}/instance_id`. On
   a Docker add-on this is `/data/instance_id`. On a standalone
   install with no explicit `data_dir`, it falls back to
   `{devices_file_parent}/instance_id`. Is the fallback sensible?
6. **Legacy `ipbuilding` custom_component on the same HA install**:
   known to coexist on the user's machine. Could that cause any
   collision in the new flow?
7. **Zeroconf name `_ipbgw`** is short and a bit cryptic — better
   trade-off than the long name, but worth a UX call-out.
